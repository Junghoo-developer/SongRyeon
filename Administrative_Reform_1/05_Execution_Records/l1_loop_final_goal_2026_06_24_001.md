# l1 loop final goal 2026-06-24 001

## 배경

이전 변경에서는 L1의 거시목표를 "에이전트 내부 운영 목표"로 정의했다.

하지만 사용자는 이 방향이 14B급 로컬 모델에게 너무 추상적이며,
거시목표가 장식 문구가 될 위험이 있다고 지적했다.

따라서 L1의 목표 정의를 다시 조정했다.

## 새 기준

L1은 철학자가 아니라 이번 L루프의 작업반장이다.

거시목표는 다음을 뜻한다.

```text
이번 L루프가 끝날 때 달성해야 하는 최종 운영 목표
```

미시목표는 다음을 뜻한다.

```text
그 최종 목표를 향해 지금 바로 수행해야 하는 중간 과정이나 행동
```

## L1에게 알려준 L루프 구조

프롬프트에 L루프의 흐름을 명시했다.

1. L1은 이번 L루프의 최종 운영 목표와 다음 중간 목표를 정한다.
2. L2는 L1 목표를 보고 검색어 또는 문서 열람 계획을 만든다.
3. 런타임은 `search_docs`, `read_artifact`, `read_doc` 같은 문서 도구를 실행한다.
4. L3는 L1의 거시/미시 목표에 비추어 달성 여부를 판단한다.
5. partial/failed 결과는 이후 continuation/retry 판단에 쓰일 수 있다.
6. node_3는 L루프가 모은 재료로 최종 답변을 쓰므로, L1 목표는 L3와 node_3가 쓸 수 있을 만큼 구체적이어야 한다.

## 기본 목표명

fallback과 fake adapter의 기본 목표는 다음으로 바꿨다.

```text
macro_goal = produce_l_loop_evidence_material_for_current_request
micro_goal = prepare_first_document_lookup_action
```

## 의미

이제 L1의 거시목표는 "근거 경계를 유지한다" 같은 추상 원칙이 아니라,
"이번 요청에 답하기 위해 L루프가 어떤 근거 재료를 확보해야 하는가"를 말해야 한다.

예를 들어 사용자가 여러 문서의 연관성 분석을 요구했다면,
거시목표는 여러 문서 후보, 읽은 문서, 부족 신호를 구분한 분석 재료 확보가 되어야 한다.

미시목표는 그 목표를 향해 L2가 첫 검색/열람 조건을 어떻게 준비할지 말해야 한다.
