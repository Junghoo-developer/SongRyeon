# task ledger v0 2026-06-23 001

## Status

Implemented and smoke-tested.

## Why This Was Built

The user and the user's father discussed that SongRyeon may eventually need a scheduler and task queue.

The important decision was not to jump directly into parallel execution.

Instead, the runtime now records the current sequential node movements as task frames and task result frames.

This gives SongRyeon a future scheduler surface without changing today's execution order.

## Implementation Summary

Added schema frames:

- `TaskFrame`
- `TaskResultFrame`

Added runtime module:

- `songryeon_core/runtime/task_ledger.py`

The task ledger converts each `NodeMovement` into:

- one `task_ledger:task_frame`
- one `task_ledger:task_result_frame`

v0 uses:

- `scheduling_policy = sequential_v0`
- `assigned_worker_id = local_sync_worker`
- first task depends on nothing
- every later task depends on the immediately previous task

## Runtime View

`qwen-chat`, `qwen-turn`, and `fake-turn --pretty` now show a short task ledger section.

Example:

```text
- task ledger: tasks=13 / results=13 / policy=sequential_v0 / worker=local_sync_worker
  - task:turn_dry_001:001 node_0:pre_route_report status=completed model=CODE:RULE_STUB
  - task:turn_dry_001:002 node_1:routing status=completed model=LLM:songryeon-all-nodes-fake-llm-adapter
  - task:turn_dry_001:003 node_0:targeted_memory_supply status=completed model=CODE:RULE_STUB
  - task:turn_dry_001:004 L:L_loop status=completed model=LLM:songryeon-all-nodes-fake-llm-adapter
```

## Important Boundary

This is not real scheduling yet.

It does not:

- run tasks in parallel
- assign work to multiple GPUs
- start worker processes
- change graph routing
- let individual nodes commit global state directly

It only records the current execution as task-shaped data.
