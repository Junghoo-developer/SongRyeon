# ORDER 086: L3 Goal Continuation And L2 Scope Revision

## 상태

이 문서는 즉시 구현 발주서가 아니다.

현재 데모에서 드러난 L루프의 검색/읽기 한계를 정리하고, 나중에 구현할 수 있는 연속 검색 구조의 선택지를 보존하기 위한 설계 메모다.

## 배경

현재 L루프는 대체로 다음 흐름으로 작동한다.

```text
L1 목표 설정
-> L2 검색어 설정
-> search_docs
-> 상위 후보 read_doc
-> L3 달성 여부 판단
-> 0
-> 1
-> 0
-> 2
```

이 흐름은 좁은 MVP에는 충분하지만, 다음 문제가 있다.

- L3가 거시/미시 목표의 성패 여부를 판단해도, 그 판단이 곧바로 추가 검색 행동으로 이어지지 않는다.
- 특정 문서를 찾지 못했거나, 부분적으로만 읽었거나, 검색 후보가 애매해도 L루프가 너무 빨리 종료될 수 있다.
- `max_read_doc_calls` 같은 예산 상한은 존재하지만, 여러 후보를 실제로 읽게 만드는 정책은 아직 약하다.
- 검색 결과 후보 순위를 재검토하거나, L2가 검색 범위를 수정하는 구조가 아직 없다.

## 핵심 문제의식

L3는 단순히 "작업이 끝났다"를 말하는 노드로 남으면 안 된다.

L3는 L1의 거시 목표와 미시 목표에 대응하여 다음을 판단해야 한다.

```text
1. 거시 목표가 달성되었는가?
2. 미시 목표가 달성되었는가?
3. 달성되지 않았다면 왜 실패했는가?
4. 실패가 검색어 문제인가, 후보 선택 문제인가, 읽기 예산 문제인가, 문서 부재 문제인가?
5. 지금 바로 추가 검색을 이어가는 것이 합당한가?
```

## 가능한 구조 선택지

### 선택지 A: L3가 1에게 돌려보내는 구조

```text
L3
-> 0
-> 1
-> 0
-> L루프 재가동
```

장점:

- 기존 원칙에 가깝다.
- 1이 중앙 라우터로서 권한을 유지한다.
- 0이 L3의 실패 사유를 정리해서 1에게 공급할 수 있다.

단점:

- 너무 무겁다.
- 단순히 2위 후보를 읽거나 검색어를 조금 바꾸면 되는 상황에서도 전체 라우팅을 다시 탄다.
- 데모용 MVP에는 과할 수 있다.

### 선택지 B: 코드가 L3 판단을 보고 제한적으로 재검색을 대신 결정하는 구조

```text
L3 달성 여부 판단
-> code controller가 partial/failed와 budget을 확인
-> 조건이 맞으면 L2 또는 read_doc 재시도
```

장점:

- 구현이 가볍다.
- L3가 직접 라우팅 권한을 갖지 않아도 된다.
- "특정 문서 못 읽음", "후보는 있으나 1개만 읽음" 같은 경우를 빠르게 보정할 수 있다.

단점:

- 코드가 LLM의 의미 판단을 대신하는 것처럼 보이면 위험하다.
- 따라서 코드는 `CODE_STATUS`와 절대 조건만 처리해야 한다.
- "왜 다시 검색하는가"의 의미 판단은 L3 또는 L2의 LLM 판단으로 남기는 것이 좋다.

### 선택지 C: L3가 L2로 가볍게 되돌리는 구조

```text
L3
-> 0
-> L2
-> search_docs/read_doc 추가 실행
-> L3 재판정
```

장점:

- `L3 -> 0 -> 1 -> 0 -> L루프`보다 가볍다.
- 실패 사유가 검색어/검색범위 문제일 때 자연스럽다.
- 1까지 돌아가지 않고도 목표 달성까지 연속 검색할 길을 열 수 있다.

단점:

- L2에게 검색 범위 수정 권한을 새로 줘야 한다.
- L2가 현재 상황을 모르고 새 검색어만 만들면 같은 실패를 반복할 수 있다.
- 따라서 L3와 L2 사이에 0이 끼어들 틈새가 필요하다.

## L2 검색 범위 수정 권한

만약 `L3 -> 0 -> L2` 구조를 허용한다면, L2는 단순히 첫 검색어를 만드는 노드가 아니라 다음 권한을 제한적으로 가져야 한다.

```text
1. 이전 검색어를 읽는다.
2. 이전 search_docs 후보를 읽는다.
3. 이전 read_doc 문서를 읽는다.
4. L3의 거시/미시 목표 달성 실패 사유를 읽는다.
5. 검색 범위를 넓히거나 좁힌 새 검색어를 만든다.
6. 필요하면 특정 후보 문서를 우선 읽도록 제안한다.
```

단, L2가 실제 도구를 직접 마음대로 실행하는 것은 아직 확정하지 않는다.

L2는 검색 계획과 검색 범위 수정을 담당하고, 실제 실행은 controller/code가 예산 안에서 수행하는 쪽이 더 안전할 수 있다.

## 0이 끼어들 틈새

L3에서 L2로 바로 보내면 L2가 현 상황을 놓칠 수 있다.

따라서 다음과 같은 0의 개입 지점이 필요하다.

```text
L3
-> 0: L3 실패/부분성공 사유 압축
-> L2: 수정 검색에 필요한 기억 공급
```

0이 L2에게 공급해야 할 정보 후보:

- L1 거시 목표
- L1 미시 목표
- L3의 거시 목표 달성 여부
- L3의 미시 목표 달성 여부
- L3의 실패/부분성공 이유
- 이미 실행한 검색어
- 이미 읽은 문서
- 아직 읽지 않은 상위 후보
- 예산 잔량
- 사용자가 특정 문서/주제를 요구했는지 여부

## 현재 결론

확실한 방향:

```text
목표 달성 전까지 제한적으로 연속 검색할 길은 열어야 한다.
```

2026-06-23 현재 추가 판단:

```text
L3는 L1 목표와 L2 검색 결과를 비판할 수 있다.
하지만 L3에게 직접 라우팅 권한을 주지는 않는다.
가벼운 L2 재검색은 당장 0을 끼우지 않고, 코드 controller가 L3 스키마에서 필요한 필드만 읽어 집행한다.
L1 목표 자체가 틀린 무거운 실패는 0과 1을 거쳐 다시 판단한다.
L2 수정 검색은 최대 3회 정도의 하드코딩 한계를 둔다.
```

다만 다음 사항은 아직 인간 결재 전이다.

```text
1. L3 비판 필드의 최종 스키마 이름을 무엇으로 할 것인가?
2. 코드 controller가 어떤 조건까지 L3 판단을 집행해도 되는가?
3. L3 -> 0 -> L2 경로를 나중에 별도 경량 루프로 열 것인가?
4. read_doc 추가 횟수와 검색 후보 수 한계를 얼마로 둘 것인가?
5. 어떤 경우에 partial로 멈추고, 어떤 경우에 failed 후 재시도할 것인가?
```

## 구현 전 금지

- L3가 `achieved`라고 말했으니 무조건 성공으로 간주하지 않는다.
- L3가 `partial/failed`라고 말했을 때 즉시 무한 재검색하지 않는다.
- L2에게 검색 범위 수정 권한을 주기 전, L2가 받아야 할 0의 상황 설명 payload를 먼저 설계한다.
- `L3 -> 0 -> 1 -> 0 -> L루프 재가동`은 원칙적으로 가능하지만, MVP에서는 너무 무거울 수 있으므로 바로 구현하지 않는다.

## 다음에 필요한 목표 지정

이 발주서를 구현하려면 먼저 다음 중 하나로 목표를 좁혀야 한다.

```text
1. L3 판정 강화만 구현한다.
2. search_top_k / max_query_attempts / max_read_doc_calls 이름과 의미를 분리한다.
3. L3 partial이면 읽지 않은 2위 후보를 한 번 더 read_doc 한다.
4. L3 failed이면 0이 실패 사유를 압축해 L2에게 재검색 조건을 준다.
```

현재 권장 첫 단계:

```text
search_top_k / max_query_attempts / max_read_doc_calls를 분리하고,
L3가 예산과 읽은 문서 수를 근거로 partial/failed를 더 정직하게 내리게 한다.
```
