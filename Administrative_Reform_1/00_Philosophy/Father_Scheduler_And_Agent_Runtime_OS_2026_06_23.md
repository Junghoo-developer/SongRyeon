# Father Scheduler And Agent Runtime OS 2026-06-23

## 상태

이 문서는 철학 계층의 임시 기록이다.

아직 영구 유지 체계, 헌법, 기능 지도, 개발 지도, 발주서로 승격되지 않았다.

이 문서는 2026-06-23 대화에서 나온 아버지의 구상을 잊지 않기 위해 적는다.

## 출처

이 구상의 핵심 출처는 아버지의 문제 제기다.

요지는 다음과 같다.

```text
송련은 노드별로 업무를 분리했다.
업무가 분리되었다면, 언젠가는 그 업무들을 배정하고 실행 순서를 관리하는 스케줄러가 필요하다.
서로 순차적으로 영향을 주지 않는 task는 동시에 돌릴 수 있고,
의존성이 있는 task는 순차적으로 돌리면 된다.
이 구조는 로컬 모델뿐 아니라 Codex, Claude, OpenAI API 같은 외부 모델을 붙였을 때 더 큰 장점이 된다.
```

이 생각은 단순한 속도 개선 아이디어가 아니다.

송련을 "LLM 호출을 순서대로 이어 붙인 프로그램"에서 "인지 작업을 운영하는 시스템"으로 확장하는 씨앗이다.

## 현재 구조에서 보이는 한계

현재 송련은 기본적으로 함수 호출형 순차 런타임이다.

대략 다음 흐름을 따른다.

```text
user
-> 0
-> 1
-> L1
-> L2
-> tools
-> L3
-> 0
-> 1
-> 0
-> 2
-> 3
-> 4
```

이 구조는 이해하기 쉽고 MVP에는 좋다.

하지만 노드가 늘어나고, 루프가 늘어나고, 모델이 여러 개가 되면 다음 문제가 생긴다.

```text
어떤 노드가 언제 실행 가능한지 중앙에서 판단하기 어렵다.
서로 독립적인 작업도 불필요하게 기다릴 수 있다.
어떤 모델 또는 GPU가 어떤 노드를 맡아야 하는지 정책화하기 어렵다.
실패, 재시도, 반려, 백그라운드 기억 정리를 한 흐름 안에 자연스럽게 넣기 어렵다.
```

## 아버지의 핵심 통찰

노드가 분리되었다면, 노드는 실행 단위가 아니라 task의 원천으로 볼 수 있다.

그렇다면 송련에는 장기적으로 다음 계층이 필요하다.

```text
Scheduler / Coordinator
Task Queue
Worker
Trace/Data Committer
Resource Policy
```

이 구조에서는 모든 일을 무조건 병렬로 돌리지 않는다.

오히려 핵심은 다음에 있다.

```text
동시에 돌려도 되는 일만 동시에 돌린다.
순차 의존성이 있는 일은 순차로 돌린다.
각 task의 입력과 출력을 명확히 한다.
최종 trace/data 반영은 중앙에서 통제한다.
```

따라서 task queue는 병렬화를 위한 장난감이 아니라, 동기 깨짐을 막기 위한 운영 구조다.

## Task Queue가 주는 의미

Task Queue 구조가 생기면 현재의 순차 실행도 그대로 표현할 수 있다.

```text
worker 1개 + task queue = 현재와 비슷한 순차 실행
worker 여러 개 + task queue = 안전하게 확장 가능한 비동기 실행
```

즉 처음부터 복잡한 멀티스레드 시스템을 만들 필요는 없다.

먼저 모든 노드 실행을 task로 감싸고, 실행은 순차로 유지할 수 있다.

그 다음 안전한 곳부터 확장하면 된다.

예시는 다음과 같다.

```text
L2가 여러 검색 후보를 만든다.
각 검색 후보는 독립적인 search_docs task가 된다.
각 read_doc 또는 read_artifact도 독립 task가 된다.
L3는 여러 결과가 모인 뒤 aggregate task로 판단한다.
```

## Agent Runtime OS 비유

이 구상은 운영체제와 닮았다.

일반 운영체제는 프로세스, 메모리, 파일, 입출력, 권한을 관리한다.

송련의 장기 구조는 인지 작업, 모델, 기억, 도구, 검수, 출력 권한을 관리한다.

대응 관계는 다음과 같다.

```text
Scheduler / Coordinator = 커널 또는 작업 스케줄러
Node task = 프로세스 또는 작업
LLM call / tool call = 시스템 자원 사용
0 = 기억과 context를 공급하는 paging 계층
1 = 실행 경로를 고르는 라우터
L loop = 문서 탐색과 근거 확보 작업군
2 = 메타정보 경계 설정 계층
3 = 사용자 출력 생성 계층
4 = 최종 검수와 출력 보안 계층
Trace/Data Store = 실행 장부와 파일시스템
심야정부 = 백그라운드 데몬 또는 야간 유지보수 작업
```

따라서 송련은 운영체제를 만드는 프로젝트는 아니지만, 장기적으로는 Agent Runtime OS에 가까운 모양을 가질 수 있다.

## 살아 움직이는 컴퓨터라는 장기 비전

아버지의 구상과 지금의 송련 구조를 합치면, 장기적으로 송련은 단순 챗봇이 아니라 살아 움직이는 컴퓨터에 가까워질 수 있다.

여기서 "살아 움직인다"는 말은 생물학적 생명을 뜻하지 않는다.

다음 성질을 가진 시스템을 뜻한다.

```text
자신의 내부 문서를 읽는다.
자신의 실행 trace를 기록한다.
자신의 실패를 검수한다.
자신의 기억을 정리한다.
자신의 다음 작업을 queue에 올린다.
필요한 모델과 도구를 배정한다.
백그라운드에서 장기기억과 구조를 관리한다.
```

이 성질이 충분히 자라면, 송련은 "사용자 입력에 반응하는 프로그램"을 넘어 "자기 내부 상태를 운영하는 인지 시스템"에 가까워진다.

이것은 인공생명이라는 말을 조심스럽게 떠올리게 한다.

다만 지금 단계에서는 인공생명이라고 단정하지 않는다.

현재의 정확한 표현은 다음과 같다.

```text
송련은 Agent Runtime OS를 향해 갈 수 있는 초기 실험체다.
이 구조가 장기기억, 자기 점검, task queue, scheduler, 백그라운드 루프를 갖추면
살아 움직이는 컴퓨터라는 비유가 점점 더 적절해질 수 있다.
```

## API 모델 연결 시 장점

이 구조는 로컬 모델보다 API 모델을 붙일 때 더 강해질 수 있다.

예를 들면 다음과 같다.

```text
node_1 router = 빠르고 싼 모델
L2 query planner = 로컬 모델 또는 싼 API 모델
L3 critic = 더 엄격한 모델
node_3 reporter = 글쓰기 좋은 모델
node_4 gatekeeper = 가장 신뢰할 수 있는 검수 모델
심야정부 = 느리지만 깊게 생각하는 백그라운드 모델
```

스케줄러가 있으면 비용, 속도, 정확도에 따라 task를 배정할 수 있다.

```text
비용 절약 모드
정확도 우선 모드
속도 우선 모드
심야정부 야간 처리 모드
학습 기록 강화 모드
```

이것은 나중에 Codex, Claude, OpenAI API, 로컬 Ollama 모델을 섞을 때 매우 중요한 장점이 된다.

## 절대 지켜야 할 원칙

Task queue와 비동기 구조를 도입하더라도 송련의 메타정보 원칙은 깨지면 안 된다.

따라서 다음 원칙을 둔다.

```text
각 task는 입력 snapshot을 받는다.
각 task는 자기 output frame만 만든다.
각 task가 직접 전역 state를 마음대로 수정하지 않는다.
trace/data commit은 중앙 Coordinator가 한다.
절대정보, 상대정보, 혼합정보는 계속 분리한다.
LLM 판단은 LLM 판단으로 기록하고, code 판단은 code 판단으로 기록한다.
병렬 실행 여부와 무관하게 최종 trace는 사람이 따라갈 수 있어야 한다.
```

이 원칙이 없으면 병렬화는 송련의 능력이 아니라 혼란이 된다.

## 당장의 구현 방향

지금 바로 복잡한 병렬 실행을 구현하지 않는다.

우선 다음 순서가 적절하다.

```text
1. 현재 순차 런타임을 학습한다.
2. TaskFrame, TaskResultFrame, SchedulerFrame 같은 최소 스키마를 구상한다.
3. 실제 실행은 순차로 유지하되, 노드 실행을 task 단위로 기록한다.
4. L loop의 search/read 계열처럼 독립성이 높은 구간부터 queue화한다.
5. 나중에 worker와 GPU/API 모델 배정을 붙인다.
```

즉 다음 목표는 "병렬화"가 아니라 "병렬화 가능한 운영 구조"다.

## 임시 결론

아버지의 스케줄러/task queue 구상은 송련의 장기 진화 방향과 잘 맞는다.

송련은 지금 노드 기반 에이전트에서 Agent Runtime OS로 확장될 수 있는 문턱에 있다.

이 문서는 그 문턱을 잊지 않기 위한 철학 기록이다.

아직 구현 명령은 아니다.

나중에 학습, 목표 설정, 발주서 작성 과정을 거쳐 신중하게 승격해야 한다.
