# L3 Critic And Lightweight Continuation 2026-06-23

## 상태

이 문서는 철학 계층의 임시 사고 기록이다.

아직 유지 체계나 헌법으로 승격하지 않는다.

## 문제의식

L3는 본점에서 비판자 성격을 가진 노드였다.

연습판에서는 현재 L3가 L1의 거시/미시 목표를 기준으로 L루프 결과의 성패를 판단하지만, L1 목표 자체를 비판하거나 검색 흐름을 다시 열 권한은 약하다.

이 상태에서는 다음 문제가 생긴다.

```text
L1 목표가 틀렸는데도 L3가 그 목표를 기준으로만 판정한다.
L2 검색이 부족했는데도 후보가 있다는 이유만으로 stop_success가 될 수 있다.
특정 문서를 못 읽었는데도 운영상 검색이 됐다는 이유로 achieved가 될 수 있다.
```

## 권한 감각

L3에게 필요한 것은 직접 라우팅 권한이 아니라 비판권이다.

L3는 다음을 말할 수 있어야 한다.

```text
L1 목표가 사용자 요청과 맞는가?
L1 목표가 L루프에 적합한가?
L2 검색이 L1 목표에 충분히 대응했는가?
읽은 문서와 검색 후보가 거시/미시 목표 달성에 충분한가?
부족하다면 검색어 문제인가, 후보 선택 문제인가, 읽기 예산 문제인가, 문서 부재 문제인가?
```

하지만 L3가 곧바로 라우팅을 실행하는 것은 아직 위험하다.

따라서 L3의 기본 권한은 다음처럼 둔다.

```text
비판한다.
이유를 쓴다.
다음 행동을 추천한다.
직접 라우팅하지 않는다.
```

## 가벼운 실패와 무거운 실패

실패는 두 종류로 나눠 보는 것이 좋다.

### 가벼운 실패

L1 목표는 대체로 맞지만, L2 검색어 또는 읽은 문서가 부족한 경우다.

예시:

```text
검색어가 너무 뭉뚱그려졌다.
1위 후보만 읽어서 특정 문서를 놓쳤다.
후보는 있었지만 충분히 읽지 않았다.
```

이 경우는 굳이 `L3 -> 0 -> 1 -> 0 -> L루프` 전체를 다시 탈 필요가 없다.

현재 MVP에서는 다음처럼 가볍게 처리하는 것이 낫다.

```text
L3
-> code controller가 L3 스키마에서 필요한 필드만 읽음
-> L2 수정 검색
-> search_docs/read_doc
-> L3 재판정
```

여기서 코드는 의미 판단을 하지 않는다.

코드는 L3가 남긴 `retry_l2` 같은 추천과 시도 횟수 같은 절대 조건만 집행한다.

### 무거운 실패

L1 목표 자체가 사용자 요청과 어긋난 경우다.

예시:

```text
사용자는 특정 문서를 찾으라고 했는데 L1이 일반 개선 목표로 잡았다.
사용자는 요약을 원했는데 L1이 개발 실행 계획으로 목표를 세웠다.
```

이 경우는 검색어만 바꿔서 해결되지 않는다.

이때는 0과 1을 거치는 무거운 경로가 필요하다.

```text
L3
-> 0: L1 목표 비판과 실패 이유 압축
-> 1: 재라우팅 판단
-> 0: 새 L1 또는 다음 대상에게 실패 맥락 공급
-> L1 재시작 또는 다른 경로
```

## 0의 위치

0은 비판자가 아니다.

0은 대상 노드에 맞춰 기억을 공급하는 역할이다.

따라서 L3가 L1을 비판했을 때 0이 할 일은 판단이 아니라 전달이다.

```text
L3의 비판 이유를 보존한다.
1이 볼 수 있게 압축한다.
다음 대상 노드가 같은 실패를 반복하지 않도록 필요한 기억을 공급한다.
```

다만 가벼운 L2 재검색에서는 당장 0을 끼우지 않아도 된다.

MVP에서는 코드가 L3 스키마에서 필요한 부분만 뽑아 L2에게 넘기는 편이 더 작고 명확하다.

## 반복 한계

연속 검색은 열어야 하지만 무한 루프가 되어서는 안 된다.

현재 감각상 가벼운 `L3 -> L2` 수정 검색은 최대 3회 정도로 하드코딩하는 것이 건강하다.

```text
max_l2_revision_attempts = 3
```

3회 안에 해결되지 않으면 더 붙잡지 않는다.

그때는 0과 1이 다시 상황을 보게 하고, 부분 보고를 할지, L1부터 다시 잡을지, 실패를 인정할지 판단하게 한다.

## 현재 철학 결론

```text
L3는 비판자다.
L3는 직접 라우터가 아니다.
가벼운 검색 실패는 코드가 L3 스키마를 집행해 L2 재검색으로 처리한다.
L1 목표 자체가 틀린 무거운 실패는 0과 1을 거쳐 다시 판단한다.
가벼운 재검색은 최대 3회로 막는다.
```

이 문서는 아직 철학이다.

구현 전에는 반드시 임시 발주서 또는 정식 발주서로 승격해야 한다.
