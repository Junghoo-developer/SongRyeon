# Raw Memory Window And Node 5 Compression Philosophy 2026-06-26

**계층**: 00 철학
**상태**: 미승격 설계 철학
**권한**: 실행 권한 없음. 향후 지도, 영구 유지 체계, 발주서로 승격 심사한다.

## 1. 문제의식

최근 대화 원문은 너무 빨리 버리면 사용자가 방금 말한 맥락을 잃는다.

반대로 모든 대화 원문을 계속 들고 있으면 0 기억공급관의 입력이 커지고, 나중에는 기억 공급이 아니라 원문 더미 전달이 된다.

따라서 SongRyeon Core의 대화 기억은 다음 두 층을 분리해서 생각한다.

```text
최근 원문 기억
-> 아직 압축하지 않고 그대로 읽을 수 있는 뜨거운 기억

압축 기억
-> 여러 턴의 원문을 근거 묶음으로 삼아 LLM이 만든 의미 요약 기억
```

## 2. 최근 원문 창 원칙

최근 원문 기억은 최대 8턴을 기본 상한으로 둔다.

다만 최근 대화 맥락이 너무 빨리 사라지지 않도록, 최근 3턴 원문은 최소 보장 창으로 본다.

초기 철학 기준은 다음과 같다.

```text
recent_raw_conversation_max_window = 8
recent_raw_conversation_min_guarantee = 3
compression_batch_size = 4
post_compression_raw_keep = 5
```

의미:

1. 최근 원문은 최대 8턴까지 그대로 들고 있는 것을 목표로 한다.
2. 9번째 원문 턴이 들어와 원문 창이 넘치려 하면, 최신 5턴은 원문으로 남긴다.
3. 그보다 오래된 4턴은 압축 후보로 분리한다.
4. 최근 3턴 원문은 특별한 비상 정책이 없는 한 압축하거나 제거하지 않는다.

예시:

```text
turn 1-8
-> 1,2,3,4,5,6,7,8을 원문으로 보관

turn 9 진입
-> 1,2,3,4를 압축 후보로 분리
-> 5,6,7,8,9는 원문으로 유지

turn 13 진입
-> 5,6,7,8을 압축 후보로 분리
-> 9,10,11,12,13은 원문으로 유지
```

이 숫자들은 영구 진리가 아니라 첫 운영 감각이다. 그러나 바꿀 때는 반드시 정책 문서 또는 발주서에서 이유를 드러낸다.

## 3. 0의 역할

0 기억공급관은 기억을 새로 쓰는 존재가 아니다.

0은 다음을 할 수 있다.

- 최근 원문 대화 좌표를 읽는다.
- TurnStateCapsule 좌표를 읽는다.
- 원문 대화와 캡슐을 `turn_id` 같은 절대정보 기준으로 대응시킨다.
- 압축이 필요해진 원문 묶음의 좌표를 압축 후보로 표시한다.
- 어떤 후보가 어떤 source trace/data에 기대는지 기록한다.

0은 다음을 하지 않는다.

- 4턴 묶음의 의미를 요약하지 않는다.
- 사용자의 감정, 의도, 장기 목표를 단정하지 않는다.
- 오래된 대화를 혼자 기억으로 승격하지 않는다.
- 압축 결과가 맞다고 승인하지 않는다.

즉 0은 기억 생성자가 아니라 기억 좌표 공급자이자 압축 후보 관리자다.

## 4. 5 기억 압축기 구상

오래된 4턴 원문 묶음을 실제 기억 요약으로 바꾸는 책임은 별도 노드가 맡는 것이 낫다.

사용자 구상 기준 이름은 다음과 같다.

```text
node_5_memory_compressor
5 기억 압축기
```

5는 LLM 노드로 설계할 수 있다.

5가 할 일:

- 0이 지정한 압축 후보 source bundle을 읽는다.
- 여러 턴을 묶어 의미 요약을 만든다.
- 요약이 어떤 turn_id, trace_id, data_id 묶음에 근거했는지 남긴다.
- 압축 결과를 곧바로 활성 기억으로 만들지 않고, 검수 대기 상태로 둔다.

5가 하면 안 되는 일:

- source bundle 밖의 기억을 끼워 넣기.
- 사용자의 숨은 의도나 감정을 확정된 사실처럼 쓰기.
- 4의 검수 없이 활성 기억으로 승격하기.
- 코드가 쓴 절대정보처럼 자기 요약을 표시하기.

## 5. 4의 승인 구상

압축 기억은 사용자가 나중에 다시 만날 가능성이 크다.

따라서 5가 만든 요약은 곧바로 장기 기억이나 활성 기억이 되면 안 된다.

4는 기억 요약 gatekeeper로 개입할 수 있다.

4가 볼 것:

- 압축 요약이 실제 source bundle에 근거하는지.
- source turn 수, source id, trace/data id가 맞는지.
- 원문에 없는 감정, 의도, 목표를 과하게 단정하지 않는지.
- 요약이 `mixed` 정보로 표시되어 있는지.
- 승인 전 상태가 active memory로 표시되지 않는지.

4가 승인하기 전의 압축 결과는 다음 상태에 가깝다.

```text
compression_candidate
pending_memory_summary
not_active_memory
```

4가 통과시킨 뒤에야 다음 단계의 기억 구조로 들어갈 수 있다.

## 6. 메타정보 분류

최근 원문 대화와 TurnStateCapsule 좌표 자체는 절대정보다.

예:

- `turn_id`
- `trace_id`
- `data_id`
- 원문 존재 여부
- 원문 문자 수
- capsule 존재 여부
- movement count

0이 만드는 압축 후보 좌표도 절대정보다.

하지만 5가 만드는 압축 요약은 혼합정보다.

이유:

```text
하나의 압축 요약
-> 여러 턴 원문과 여러 trace/data record의 source bundle
-> 하나의 절대정보 record에 일대일 대응시키기 부적절함
```

따라서 압축 요약은 다음 원칙을 따른다.

```text
info_class = mixed
source_mode = source_bundle
semantic_judgement_status = ran
generated_by = LLM node 5
approval_status = pending | approved | rejected
approved_by = node_4 또는 not_run
```

## 7. 지금 당장 하지 않는 것

이 문서는 발주서가 아니다.

지금 당장 구현하지 않는다.

특히 다음은 아직 열지 않는다.

- vector DB

- 계층형 검색

- 0의 의미 요약 생성

## 8. 향후 발주서 후보

이 철학이 구현 단계로 내려가려면 최소 세 발주로 쪼개는 것이 안전하다.

```text
ORDER_107_RAW_MEMORY_WINDOW_AND_COMPRESSION_CANDIDATE_POLICY_V0
```

최근 원문 창의 최대 8턴, 최소 3턴 보장, 9턴 진입 시 5턴 유지 + 4턴 압축 후보 분리 정책을 스키마와 smoke로 고정한다.

```text
ORDER_108_NODE5_MEMORY_COMPRESSOR_V0
```

5 기억 압축기 LLM 노드를 만들되, 압축 결과는 `pending_memory_summary`로만 저장한다.

```text
ORDER_109_NODE4_MEMORY_SUMMARY_GATE_V0
```

4가 5의 압축 요약을 검사하고, 통과한 요약만 활성 기억 후보로 승격할 수 있게 한다.

## 9. 임시 결론

최근 원문 기억은 최대 8턴까지 들고 간다.

9턴째가 오면 최신 5턴을 남기고, 오래된 4턴은 압축 후보가 된다.

0은 좌표를 찍는다.

5는 요약한다.

4는 승인한다.

코드는 숫자, id, schema, 상태만 확정한다.

의미 요약은 LLM이 쓰고, 그 요약은 여러 source에 기대므로 혼합정보로 남긴다.
