# Admin Rules v0

## 기본 원칙

1. 사용자가 설계를 말하는 동안에는 코드 작업을 하지 않는다.
2. 사용자의 말을 먼저 구조화하고, 필요하면 문서로 정리한다.
3. 코드 작업은 발주서 또는 명시적인 구현 요청 이후에만 시작한다.
4. 구현은 작은 단계로 나누고, 각 단계의 목적과 결과를 설명한다.

## 문서 권한

### 철학

애매한 생각, 초기 문제의식, 아직 미분류된 개념을 보관한다. 실행 권한은 없다.

### 영구 유지 체계

문서 관리, 에이전트 협업, 작업 중지 조건, 승격 절차를 다룬다.

### 헌법

큰 방향 전환을 기록한다. 이전 구조와 새 구조를 비교하고, 앞으로의 비전을 제시한다.

### 지도

구조를 이해하고 학습하기 위한 문서다. 기능 지도는 현재 구조를, 개발 지도는 구현 순서를 다룬다.

### 발주서

실제로 코딩할 작업을 작게 쪼갠 지시서다.

### 실행 기록

대화 기록, 턴 회고, 리포트, 검수 결과를 보관한다. 현재 명령으로 취급하지 않는다.

## 승격 규칙

철학 문서의 내용이 반복해서 필요해지면 지도 후보로 승격한다.  
지도에서 구현 순서가 명확해지면 발주서로 승격한다.  
발주서가 실행되면 결과는 실행 기록으로 남긴다.

## 스키마와 데이터 설계

스키마, state, trace, memory packet 같은 데이터를 다룰 때는 반드시 절대 정보, 상대 정보, 혼합 정보를 분리하여 명시한다.

절대 정보는 먼저 확정할 수 있다. 상대 정보와 혼합 정보는 관련 노드 설계, 프롬프트 설계, 출력 스키마 설계와 병행하거나 연계하여 확정한다.

세부 규칙은 [SCHEMA_METAINFO_POLICY_v0.md](SCHEMA_METAINFO_POLICY_v0.md)를 따른다.

특히 코드가 상대정보나 혼합정보의 의미 문장을 직접 새로 쓰는 것을 금지한다. 코드는 원문 복사, 출처 연결, 상태 표시, 스키마 검증, 절단 여부 표시만 수행할 수 있다.
