# Schema Metainfo Policy v0

## Status

This document is a maintenance-system rule.

It is not an implementation order, but all future schema, runtime, node, prompt, and report work should follow it unless a newer maintenance-system document replaces it.

## Core Thesis

SongRyeon must not let users confuse code output, LLM output, tool output, document text, and human-written text.

Every user-visible or node-visible information item should make clear:

1. who or what generated it,
2. where it came from,
3. whether it is absolute, relative, or mixed information,
4. whether semantic judgement has actually run,
5. whether the text was copied or newly generated.

## Information Classes

### Absolute Information

Absolute information is information that code can record, check, copy, or enforce inside the running system.

Absolute information is not world-truth. It is system-checkable reference information.

Examples:

- a `data_id` exists,
- a `trace_id` exists,
- a model call happened,
- a model returned a raw text,
- a file exists,
- a document was read,
- a tool returned a score,
- a schema validation passed or failed,
- a payload field has a certain value,
- a runtime policy flag was enabled.

Only code may write absolute information.

### Relative Information

Relative information is interpretation, judgement, summary, intention, explanation, or natural-language meaning produced from one specific absolute information item.

The key condition is one-to-one grounding:

```text
one relative claim
-> one identifiable absolute source record or field
```

The system can check that the source record exists and that the relative claim was generated, but it cannot check the semantic truth of the claim itself.

If a semantic claim cannot honestly be grounded in one specific absolute information item, do not classify it as relative information. Classify it as mixed information with a source bundle.

Examples:

- user intent interpretation,
- route justification,
- goal reason,
- document summary,
- answer draft,
- semantic sufficiency judgement,
- claim about what a document means,
- claim about whether a goal was meaningfully achieved.

Relative information may be produced by:

- a user,
- a human-authored document,
- an LLM,
- a future reviewer node.

Code must not write relative information as if it were semantic judgement.

### Mixed Information

Mixed information is not absolute information plus relative information becoming certain.

Mixed information is also not "any relative information that has a source link."

Mixed information is semantic information generated from absolute sources when the system cannot honestly assign the claim to one specific absolute information item.

The key condition is source-bundle grounding:

```text
one mixed claim
-> multiple absolute source records
or -> a source bundle where one-to-one grounding is impossible or inappropriate
```

The source bundle does not prove the claim true. It only records the generation path and makes later review possible.

Do not force false one-to-one provenance. If a claim was generated by comparing, summarizing, balancing, or interpreting several records together, classify it as mixed information.

Example source modes:

```text
info_class: relative
source_mode: direct_field
source_data_ids:
  - tool_result:read_doc:trace_000019
source_field_path: text
```

```text
info_class: mixed
source_mode: source_bundle
source_data_ids:
  - L1:goal_frame
  - tool_result:search_docs:trace_000014
  - tool_result:read_doc:trace_000019
claim_alignment: not_line_mapped
```

## Code No-Paraphrase Rule

Code does not write semantic text.

Code may copy text.

Code may:

- copy user input,
- copy LLM raw output,
- copy document text,
- copy tool payload fields,
- copy schema/status labels,
- attach IDs,
- attach provenance,
- mark validation status,
- truncate text,
- sort records,
- filter records.

Code must not:

- invent a natural-language reason,
- summarize a document,
- interpret user intent,
- explain why a goal is meaningful,
- claim semantic sufficiency,
- call an operational check a semantic judgement,
- make a document claim sound true merely because a source exists.

If code copies text, it should record:

```text
copied_from: source_data_id.field_path
selection_by: CODE
selection_method: first_heading | first_result | configured_limit | exact_field
truncated: true | false
```

If code truncates text, it must not hide that truncation happened.

## Code May Write Status Labels

Code may write controlled labels because they are operational state, not semantic prose.

Allowed examples:

- `passed`
- `failed`
- `not_run`
- `missing_source`
- `schema_failed`
- `CODE:RULE_STUB`
- `CODE:POLICY_STUB`
- `CODE:OPERATION_CHECK`
- `LLM:qwen3:14b`
- `TOOL_RESULT`
- `DOCUMENT_EXTRACT`

When a code-generated status can be mistaken for a semantic judgement, prefix or label it clearly.

Preferred:

```text
operation_achieved
CODE:OPERATION_CHECK
LLM_SEMANTIC=not_run
```

Avoid:

```text
goal achieved
the answer is sufficient
the document is relevant
```

unless those words are copied from an LLM or human source and marked as such.

## LLM Output Rule

LLM-generated text should be preserved as LLM output, not rewritten by code.

Code may validate:

- JSON parse status,
- schema status,
- required fields,
- source IDs,
- token/size limits,
- whether the model call exists.

Code must not silently rewrite the meaning of LLM text.

If the system normalizes an LLM output for schema compatibility, it must record the normalization or expose that only a schema-compatible subset survived.

## Tool and Document Output Rule

Tool output is not LLM output.

Document text is not code output.

A copied document excerpt should be labeled as document extract or copied text.

A tool score should be labeled as tool-returned score, not truth or relevance by itself.

Example:

```text
TOOL_RESULT/search_docs score=0.344462
```

means:

```text
the tool returned this score
```

It does not mean:

```text
this document is truly the best answer
```

## Node Honesty Rule

If a node is planned as an LLM node but currently implemented by code, it must be labeled as a stub.

Examples:

```text
node_0: CODE:RULE_STUB, LLM_SUMMARY=not_run
node_1: CODE:RULE_OR_POLICY_STUB, LLM_ROUTER=not_run
L1: RULE_STUB, LLM_GOAL=not_run
L3: CODE:OPERATION_CHECK, LLM_SEMANTIC=not_run
node_3: CODE/RENDERER, LLM_REPORTER=not_run
```

Only a node that actually calls an LLM may label its generated semantic text as LLM output.

## Node 2, Node 3, Node 4 Governance

### Node 2

Node 2 manages metainfo boundaries and mixed information.

If node 2 creates mixed information with an LLM, it must mark it as LLM-generated mixed information and record source links.

Node 2 must not let source-linked mixed information become absolute information.

### Node 3

Node 3 creates user-facing report drafts.

Node 3 must not hide information status.

If node 3 is still a code renderer, it must say so.

If node 3 becomes an LLM reporter, its generated report text must be marked as LLM output and passed to node 4 before final release.

### Node 4

Node 4 is a pre-output gatekeeper, not a post-output auditor.

Node 4 may reject a draft and return the turn to another node or loop.

Node 4 may reject for:

- missing source,
- provenance mismatch,
- information-status mismatch,
- unsupported claim,
- overclaiming,
- schema violation,
- renderer hiding uncertainty,
- need for more evidence.

Node 4 does not become an all-knowing truth judge.

If node 4 uses LLM review, that review is itself LLM-generated relative or mixed information.

## Return Loop Principle

When node 4 rejects, it should name the return target.

Examples:

- insufficient evidence -> return to `L`,
- routing mismatch -> return to `1`,
- missing memory -> return to `0`,
- mixed information problem -> return to `2`,
- report rendering problem -> return to `3`.

## Required Runtime Labels

Every runtime output intended for humans should expose at least:

```text
generated_by
info_class
source_data_ids
semantic_judgement_status
```

When exact source alignment is not possible:

```text
source_mode: source_bundle
claim_alignment: not_line_mapped
```

## Development Rule

Before adding a new LLM node, define:

1. what absolute information code will write,
2. what relative information the LLM may write,
3. how mixed information will link to sources,
4. what code is allowed to copy,
5. what node 4 can reject,
6. what must be displayed as `not_run` while the LLM node is not yet implemented.

## Current Project State

As of this policy version:

- L2 query planning may use Qwen.
- node 0 is still `CODE:RULE_STUB`.
- node 1 is still `CODE:RULE_OR_POLICY_STUB`.
- L1 is still `RULE_STUB`.
- L3 is still `CODE:OPERATION_CHECK`.
- node 3 is still `CODE/RENDERER`.
- node 4 is not implemented yet.

This must remain visible in runtime output until those nodes are genuinely upgraded.
